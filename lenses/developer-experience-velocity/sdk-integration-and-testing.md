# SDK Integration & Testing

Patterns for integrating LaunchDarkly into a codebase well, and for testing the result with confidence. This page covers the day-to-day practice of writing code that uses flags.

---

## SDK initialization

### Pattern 1 — One client per process, shared

For server-side SDKs, the client is a long-lived singleton initialized once per process. Application code accesses the same instance.

```pseudocode
// At process startup
ldclient = LDClient.init(sdk_key, config)

// Anywhere in the application
result = ldclient.boolVariation(flag_key, context, fallback)
```

Don't re-initialize per request. Don't create multiple clients. The SDK is designed for the singleton pattern.

### Pattern 2 — Wrapper library for the team's defaults

For larger teams, wrap the SDK in a small internal library that encodes the team's standards:

- Timeouts and configuration the team has decided on.
- Logging and metrics emission for every evaluation.
- A typed interface for flag access (see Type Safety below).
- Sensible fallback handling.

```pseudocode
// Team wrapper
class FlagClient:
  def __init__(self, sdk_key, config_overrides={}):
    self.client = LDClient.init(sdk_key, default_config | config_overrides)
    self.metrics = MetricsRegistry.get()

  def bool(self, key: FlagKey, context, default: bool) -> bool:
    start = monotonic()
    try:
      result = self.client.boolVariation(key.value, context, default)
      self.metrics.histogram("ld.eval.duration", monotonic() - start, tags={"flag": key.value})
      return result
    except Exception as e:
      self.metrics.counter("ld.eval.error", tags={"flag": key.value})
      logger.error(f"LD evaluation failed for {key.value}", exc_info=e)
      return default
```

The wrapper's value scales with team size — for a small team, the bare SDK is fine.

### Pattern 3 — Initialization timeout

Configure an explicit init timeout. The application proceeds with fallback values after the timeout. SDK initialization continues in the background.

```pseudocode
config = LDConfig(
  sdk_key=sdk_key,
  start_timeout_seconds=5,  // explicit; don't rely on defaults silently
)
ldclient = LDClient.init(config)
```

The team that doesn't bound init couples application startup to LaunchDarkly availability.

## Type safety and codegen

### Pattern 4 — Generated flag-key constants

Flag keys referenced as string literals are typo-prone. Generate constants from the LaunchDarkly account:

```pseudocode
// Generated by build tooling; do not edit by hand.
package flags

class FlagKey:
  CHECKOUT_V2 = "checkout-v2"
  DASHBOARD_BETA = "dashboard-beta"
  AI_MODEL_VERSION = "ai-model-version"
```

The build pipeline pulls the current flag inventory from the LaunchDarkly REST API and generates the constants file. Engineers reference `FlagKey.CHECKOUT_V2`, not the string.

**Benefits:**

- IDE autocomplete on flag keys.
- Refactor-safety when keys are renamed.
- Build-time errors for references to archived flags.
- Visibility: the file is a complete inventory of flags in code.

### Pattern 5 — Typed flag returns

For multivariate flags, type the return value:

```pseudocode
// Generated from the flag's variations
class CheckoutVersion(Enum):
  V1 = "v1"
  V2 = "v2"
  V3 = "v3"

result = flag_client.string(FlagKey.CHECKOUT_VERSION, context, CheckoutVersion.V1.value)
match CheckoutVersion(result):
  case CheckoutVersion.V1: ...
  case CheckoutVersion.V2: ...
  case CheckoutVersion.V3: ...
```

The match is exhaustive; the compiler / type-checker complains when new variations exist that aren't handled.

## Local development

### Pattern 6 — Dev environment in LaunchDarkly

Most teams set up a `development` (or `dev`, `local`) environment in their LaunchDarkly project alongside `staging` and `production`. The dev environment has:

- Realistic flag values (matched to current production for stability).
- Looser permissions (engineers can change flags freely).
- A separate SDK key, scoped to dev only.
- Optionally: synthetic test data for personal-cohort testing.

Local development connects to this environment. Each engineer sees their own changes; no one's experiments interfere with another's local work.

### Pattern 7 — Offline mode for tests and air-gapped scenarios

For unit tests and CI:

```pseudocode
config = LDConfig(
  offline=true,  // SDK never calls LaunchDarkly
)
ldclient = LDClient.init(config)

// All evaluations return the fallback values
```

For offline-mode tests, the test sets up the context and passes the expected fallback. The test exercises the application's fallback-path behavior.

For local development against a fixed flag dataset (e.g., demo environments, training environments):

```pseudocode
// File-data source: SDK reads flag values from a local JSON file
config = LDConfig(
  data_source=FileDataSource("./test-flags.json"),
)
```

The team distributes the JSON file with the codebase or generates it from production exports.

## Testing patterns

### Pattern 8 — Unit test the fallback path

Every code path gated by a flag has at least two unit tests: one with the flag enabled, one with it in the fallback state.

```pseudocode
def test_checkout_renders_v2_when_flag_enabled():
  client = mock_flag_client(checkout_v2=True)
  result = checkout(client=client, ...)
  assert result.renders_v2

def test_checkout_renders_v1_when_flag_disabled():
  client = mock_flag_client(checkout_v2=False)  // the fallback path
  result = checkout(client=client, ...)
  assert result.renders_v1
```

The mock fixture provides flag values; production code reads them via the same wrapper.

### Pattern 9 — Integration test against the real SDK in offline mode

For integration tests, run the full application with the SDK in offline mode. Assert that the application's fallback behavior is correct end-to-end.

This catches the case where the unit tests stub the flag client but the integration is wired differently.

### Pattern 10 — Snapshot tests for UI

For UI applications, snapshot tests render the affected surface with each variation of the flag:

```pseudocode
def test_dashboard_renders_correctly_for_each_variation():
  for variation in DashboardVersion:
    snapshot(render_dashboard(variation=variation))
```

Visual regressions are caught at the moment they're introduced.

### Pattern 11 — Multi-variation experiment test

For multivariate flags or experiments, every variation has at least one test exercising it. Adding a new variation requires adding (or updating) the tests.

The team that ships a new variation without testing it for the first time discovers it in production.

## IaC for LaunchDarkly resources

### What belongs in IaC

The team's IaC (Terraform / Pulumi / OpenTofu) manages:

- **Projects and environments** — they shouldn't be created ad-hoc.
- **Foundational segments** — `eu-tenants`, `enterprise-plan`, `beta-cohort` — the segments that many flags reference.
- **Standard Release Pipelines** — the team's vetted rollout pattern, reused across many flags.
- **Custom roles** — the team's access policy.
- **Teams** (via SCIM, but the IaC layer ensures the structure is consistent).
- **Webhook integrations** — Slack, Datadog, etc.

These resources are slow-changing, multi-team-relevant, and benefit from reproducibility.

### What doesn't belong in IaC

- **Day-to-day flag flips and rollout-progression changes.** Too high-velocity; IaC would slow them down.
- **Customer-success-driven overrides.** Operational, not architectural.
- **Experiment configurations during active experiments.** These change frequently as the team learns.

The rule: IaC for the structure, UI for the operation.

### IaC example (Terraform)

```hcl
resource "launchdarkly_project" "main" {
  key  = "main-product"
  name = "Main Product"

  tags = ["managed-by-iac"]

  environments {
    key   = "production"
    name  = "Production"
    color = "DC143C"
  }

  environments {
    key   = "staging"
    name  = "Staging"
    color = "B0C4DE"
  }
}

resource "launchdarkly_segment" "enterprise_customers" {
  project_key = launchdarkly_project.main.key
  env_key     = "production"
  key         = "enterprise-customers"
  name        = "Enterprise Customers"

  rules {
    clauses {
      attribute = "plan"
      op        = "in"
      values    = ["enterprise"]
    }
  }
}
```

PRs to the IaC repo are reviewed; changes are deployed through the team's deployment pipeline. The IaC repo is the source of truth for the structure.

## CI/CD integration

### Pattern 12 — Flag-key lint in CI

A CI step verifies every flag-key reference in code matches an existing flag in LaunchDarkly. References to archived flags fail the build.

This is usually implemented via [LaunchDarkly Code References](https://launchdarkly.com/docs/home/code/code-references) running in CI plus a custom check that fails on references to archived flags.

### Pattern 13 — "This PR adds a flag" surfacing

When a PR adds a new flag-key reference, the CI step surfaces it: a comment on the PR, a label, a required field in the PR description.

The reviewer asks: "is the owner set? is the type tag set? is the retirement plan stated?"

### Pattern 14 — Deploy markers

Every deploy emits an event to LaunchDarkly (via webhook or via the audit log integration). The event appears on dashboards alongside flag changes. The team can correlate "we deployed X" with "the flag's behavior changed."

### Pattern 15 — Ephemeral environments

Some teams spin up per-PR ephemeral environments. The LaunchDarkly integration: the ephemeral environment uses a copy of the development environment's flag values, scoped per-PR. The PR's preview deployment evaluates against its own scoped state.

## Onboarding

### Pattern 16 — The first-week LaunchDarkly task

A new engineer's first-week task list includes a hands-on LaunchDarkly task: create a flag, integrate it in a small change, ship it through the standard rollout, archive it after. The task isn't about the feature; it's about the workflow.

The team's documentation supports this: a short "how to add a flag" walkthrough that maps to the team's specific patterns (wrapper library, IaC layer, CI checks, naming conventions).

### Pattern 17 — Pair-programming with a LaunchDarkly-experienced engineer

For the first non-trivial flag the new engineer adds, they pair with someone who's done it before. The pairing covers naming, fallback selection, testing patterns, and the rollout strategy.

This pays off in the avoided bugs of the new engineer's first months.

### Pattern 18 — A short, current internal doc

A single document, kept current, that says:

- "How we use LaunchDarkly here."
- The patterns and conventions specific to this team.
- Links to the wrapper library, IaC repo, codegen tooling.
- Common pitfalls and how to avoid them.

This is the document new engineers read first. It's also the document existing engineers reference under pressure.

## A minimum-viable DX setup checklist

For a team using LaunchDarkly seriously, the DX baseline:

- [ ] SDK wrapper or typed access layer.
- [ ] Generated flag-key constants.
- [ ] Dev environment in LaunchDarkly with realistic data.
- [ ] Offline-mode tests in CI for the fallback path.
- [ ] Code References running on every relevant repo.
- [ ] IaC for foundational resources.
- [ ] CI lint for archived-flag references.
- [ ] "How we use LaunchDarkly" doc that's actually current.
- [ ] First-week onboarding task for new engineers.

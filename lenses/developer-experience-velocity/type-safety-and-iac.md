# Type Safety & IaC

A deeper look at the two patterns that pay outsized DX dividends: making flag access type-safe in code, and managing foundational LaunchDarkly resources as infrastructure.

---

## Why type safety for flags matters

Without typed access, a flag reference is a string literal. Strings have several DX-hostile properties:

- **No IDE help.** Auto-complete doesn't surface available flag keys. Engineers either remember keys or look them up.
- **No refactor support.** Renaming `checkout-v2` to `checkout-v2-experiment` requires a find/replace across the codebase. Mistakes are silent.
- **No build-time check.** A typo (`checkut-v2`) compiles cleanly. At runtime, the evaluation returns the fallback. The team finds out from a user, a metric, or an audit.
- **No archival awareness.** When a flag is archived, code references continue to compile. The evaluation returns the fallback forever.

Typed access fixes all four.

---

## Approaches to typed access

Different ecosystems have different idioms:

### Static-language: codegen + enum/class

```kotlin
// Generated by build tooling from the LaunchDarkly API
enum class Flag(val key: String) {
    CHECKOUT_V2("checkout-v2"),
    DASHBOARD_BETA("dashboard-beta"),
    AI_MODEL_VERSION("ai-model-version"),
}

// Usage
val enabled = flagClient.bool(Flag.CHECKOUT_V2, context, default = false)
```

The codegen runs on a schedule (or on PR open) and produces the enum/class. Engineers don't edit it.

### Dynamic-language: typed wrapper + linting

```python
# A typed wrapper
class FlagClient:
    def checkout_v2(self, context, default=False) -> bool:
        return self._client.boolVariation("checkout-v2", context, default)

    def dashboard_beta(self, context, default=False) -> bool:
        return self._client.boolVariation("dashboard-beta", context, default)
```

The wrapper exposes one method per flag. Each method's return type is correct for the flag's variation type. Codegen produces this wrapper.

### Lighter-weight: a constants file + lint

For teams that don't want full codegen, a constants file maintained by hand (or partially by hand):

```python
# flag_keys.py
class FlagKeys:
    CHECKOUT_V2 = "checkout-v2"
    DASHBOARD_BETA = "dashboard-beta"

# Usage
enabled = client.boolVariation(FlagKeys.CHECKOUT_V2, context, False)
```

Plus a lint rule that fails CI if `boolVariation` is called with a string literal not in the constants file.

This is less robust than full codegen but lower-cost. Suitable for small teams.

## Codegen mechanics

### The pipeline

1. **A scheduled job** (or on-PR hook) calls the LaunchDarkly REST API to fetch the current flag inventory.
2. **The generator** transforms the inventory into the team's typed access format.
3. **The output** is committed to the repo (so PRs can reference flags that have been created in LaunchDarkly).
4. **CI verifies** the codegen output is current (re-running it produces no diff).

### What the generator produces

For each flag:

- A constant or enum value for the key.
- A typed method on the wrapper (if using the wrapper pattern).
- Documentation comments from the flag's description.
- Tags as annotations or attributes (so the type system can surface flag metadata).

### Handling archived flags

When a flag is archived in LaunchDarkly, the next codegen run should either:

- **Remove the constant** — breaking references in code; CI fails until the team removes the references.
- **Mark it deprecated** — emitting a deprecation warning; references still compile but the team is alerted.

Most teams choose the deprecation route — sudden breakage is too disruptive. The deprecation has a quarterly cleanup window.

### Handling new flags

Codegen typically runs on PR open. When a new flag is added in LaunchDarkly, the PR using it triggers a codegen run that adds the constant. The PR includes both the application change and the regenerated constants file.

For teams creating flags via IaC, the constants file is generated as part of the IaC apply.

## IaC for LaunchDarkly: depth

The boundary between IaC and UI is the most important DX decision in larger LaunchDarkly deployments.

### What goes in IaC: examples and rationale

**Projects.**
- Reasoning: project structure should be reproducible. Creating a new project ad-hoc creates governance debt.
- IaC ensures the project has the right environments, the right tags, the right team assignments from day one.

**Environments.**
- Reasoning: every project's environment structure should be consistent. Naming, color coding, default policies — all reproducible.

**Foundational segments.**
- Reasoning: segments like `eu-tenants`, `enterprise-customers`, `beta-program-q4-2026` are organizational concepts that many flags reference. They benefit from reproducibility.
- Caveat: segments whose membership changes frequently (membership-as-data) shouldn't be in IaC. The structure of the segment can be; the members shouldn't be.

**Custom roles.**
- Reasoning: access control benefits massively from IaC. Diffs to roles are reviewable; permissions don't drift; audit is trivial.

**Standard Release Pipelines.**
- Reasoning: the team's vetted rollout pattern is a reusable artifact. Building it once in IaC and applying it consistently saves significant time.

**Webhook integrations.**
- Reasoning: integration drift is a real problem. IaC keeps integrations declarative and recoverable.

### What stays in the UI

**Day-to-day flag flips and rollout progressions.**
- Reasoning: high velocity. The IaC overhead would slow these down meaningfully.

**Per-customer override segment membership.**
- Reasoning: operational; CS team needs to update without an IaC PR.

**Experiment configurations.**
- Reasoning: experiments change frequently during their lifecycle. IaC would create churn.

**Schedule-driven changes.**
- Reasoning: the team's scheduled-change capability in LaunchDarkly is the right tool, not IaC.

### Boundary cases

**Individual flag definitions.**
- Some teams put every flag in IaC. Others put none. Most put *some* (foundational, long-lived flags) and leave others (experimental, short-lived) in the UI.
- Rule of thumb: if the flag will exist for years and be referenced across teams, IaC. If it'll be archived in months, UI.

**Targeting rules.**
- Stable rules (entitlement targeting) can live in IaC. Dynamic rules (rollout progressions, override exceptions) shouldn't.

## Terraform provider patterns

LaunchDarkly publishes a Terraform provider. The patterns most teams converge on:

### Pattern A — Module per project

```hcl
module "main_product" {
  source = "./modules/launchdarkly-project"

  project_key = "main-product"
  project_name = "Main Product"

  environments = {
    production = { name = "Production", color = "DC143C" }
    staging    = { name = "Staging",    color = "B0C4DE" }
    development = { name = "Development", color = "228B22" }
  }
}
```

The module handles the project's standard shape. New projects use the module; consistency is automatic.

### Pattern B — Custom roles in their own file

```hcl
resource "launchdarkly_custom_role" "proposer_prod" {
  key = "proposer-prod"
  name = "Production Proposer"
  description = "Can propose flag changes in production; cannot approve own changes."

  policy_statements {
    actions = ["updateOn", "updateFallthrough", "updateRules"]
    effect = "allow"
    resources = ["proj/main-product:env/production:flag/*"]
  }

  // ... more policy statements
}
```

The roles file is the security team's source of truth. PRs to it are reviewed by security.

### Pattern C — Foundational segments

```hcl
resource "launchdarkly_segment" "enterprise_tier" {
  project_key = "main-product"
  env_key     = "production"
  key         = "enterprise-tier"
  name        = "Enterprise Tier Customers"

  // Membership is rule-driven, not hand-maintained
  rules {
    clauses {
      attribute = "plan"
      op        = "in"
      values    = ["enterprise"]
    }
  }
}
```

Segments whose membership is derivable belong in IaC. Membership-as-list segments live in the UI.

### Pattern D — Release Pipelines (Enterprise / Guardian)

```hcl
resource "launchdarkly_release_pipeline" "standard_rollout" {
  project_key = "main-product"
  key         = "standard-rollout"
  name        = "Standard Customer-Facing Rollout"

  // ... phases, gates, approvals
}
```

The team's standard rollout is a reusable artifact, applied to flags via the LaunchDarkly UI or via IaC.

## CI for IaC

The IaC repo's CI:

- **terraform plan** on every PR — surfaces what would change.
- **policy checks** — does the change pass the team's policy (allowed actions, expected naming, required tags)?
- **review requirements** — security review for role changes; platform review for project / environment changes.
- **apply on merge** — to a non-production LaunchDarkly project first, then to production after verification.

The IaC repo is treated like any other production-bound code repository.

---

## Drift handling

IaC's promise of reproducibility breaks when someone makes a change in the UI that the IaC doesn't capture. The patterns:

### Approach 1 — IaC is authoritative

Periodic `terraform plan` (or equivalent) shows any drift. The team resolves: either incorporate the UI change into IaC, or revert it.

### Approach 2 — Some resources are IaC-managed, others are UI-managed

The IaC explicitly declares "I manage these resources; everything else is hands-off." Drift detection only applies to managed resources.

For most teams, Approach 2 is the right balance. The boundary between IaC-managed and UI-managed is itself a documented decision.

---

## A minimum-viable IaC setup

For a team adopting IaC for LaunchDarkly:

- [ ] One IaC repo (or one section of an existing repo) for LaunchDarkly resources.
- [ ] Projects, environments, and custom roles managed in IaC.
- [ ] Foundational rule-driven segments in IaC.
- [ ] At least one Release Pipeline template in IaC (Enterprise / Guardian).
- [ ] CI for IaC: plan on PR, apply on merge.
- [ ] Documented boundary between IaC-managed and UI-managed resources.
- [ ] Quarterly drift review.
